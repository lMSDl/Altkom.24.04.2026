# Test Failures due possible code bugs

## Hangman.UnitTests.csproj - Run_DisplaysPlayAgainPrompt_AfterEachRound
- **Confidence**: High
- **Test File**: Hangman.UnitTests\GameEngineTests.cs
- **Bug Location**: Hangman\GameEngine.cs@34 and @114

### Analysis
The GameEngine class is tightly coupled to Console operations, specifically Console.Clear(), which throws IOException in test environments even when Console.SetIn/SetOut are used. This is a design issue where the class has a hard dependency on Console that prevents unit testing. The test itself is correctly written - it properly redirects console I/O and provides valid input. The issue is that Console.Clear() requires an actual console buffer to exist, which is not available during automated test execution.

### Suggested Fix
To make GameEngine testable, refactor it to depend on an abstraction for console operations. Create an interface like `IConsoleWrapper` with methods for Clear(), WriteLine(), Write(), ReadLine(), etc. Inject this interface into GameEngine's constructor. Provide a real implementation that wraps Console for production use, and a test implementation (or mock) that doesn't throw for Clear(). This follows the Dependency Inversion Principle and makes the code testable.

Example interface:
```csharp
public interface IConsoleWrapper
{
    void Clear();
    void WriteLine(string value);
    void Write(string value);
    string? ReadLine();
    ConsoleColor ForegroundColor { get; set; }
    void ResetColor();
    // other Console members as needed
}
```

Then modify GameEngine constructor:
```csharp
private readonly IConsoleWrapper _console;
public GameEngine(WordRepository repository, IConsoleWrapper console)
{
    _repository = repository;
    _console = console;
}
```

Replace all `Console.` calls with `_console.` throughout GameEngine class.

## Hangman.UnitTests.csproj - Run_WithUserChoosingNotToPlayAgain_ExitsAfterFirstRound
- **Confidence**: High
- **Test File**: Hangman.UnitTests\GameEngineTests.cs
- **Bug Location**: Hangman\GameEngine.cs@35,124

### Analysis
The test is skipped because the GameEngine.Run() method calls Console.Clear() multiple times (in ChooseLanguageAndCategory at line 35 and in Render at line 124), which throws IOException in test environments where no actual console window is available. This is a design issue in the production code - the GameEngine class is tightly coupled to the Console class, making it untestable. Even though the test redirects Console input/output using Console.SetIn() and Console.SetOut(), Console.Clear() bypasses these redirections and attempts to clear the actual console buffer, which fails when running in a test host environment. This is not a test bug - the test is correctly written to verify the expected behavior. The issue is that the production code is not designed to be testable.

### Suggested Fix
The GameEngine class needs to be refactored to abstract Console operations, making it testable. There are several approaches:

1. **Create an IConsole abstraction** and inject it:
```csharp
public interface IConsole
{
    void Clear();
    void Write(string value);
    void WriteLine(string value);
    string? ReadLine();
    ConsoleColor ForegroundColor { get; set; }
    void ResetColor();
    Encoding OutputEncoding { get; set; }
}

public class SystemConsole : IConsole
{
    public void Clear() => Console.Clear();
    public void Write(string value) => Console.Write(value);
    public void WriteLine(string value) => Console.WriteLine(value);
    public string? ReadLine() => Console.ReadLine();
    public ConsoleColor ForegroundColor { get => Console.ForegroundColor; set => Console.ForegroundColor = value; }
    public void ResetColor() => Console.ResetColor();
    public Encoding OutputEncoding { get => Console.OutputEncoding; set => Console.OutputEncoding = value; }
}
```

Then modify GameEngine to accept IConsole in the constructor and use it instead of the static Console class throughout.

2. **Simpler approach**: Wrap Console.Clear() calls in try-catch blocks to suppress IOException:
```csharp
private static void ClearConsole()
{
    try
    {
        Console.Clear();
    }
    catch (IOException)
    {
        // Ignore in test environments
    }
}
```
Then replace all `Console.Clear()` calls with `ClearConsole()`. This is a minimal change that would make the code testable without major refactoring.
